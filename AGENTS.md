# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

- Add durable project-specific notes here as they are discovered through real work.
- Setup, commands, and request flows: `README.md` and `docs/ARCHITECTURE.md` are authoritative. `npm test` / `npm run build` gate every change.
- Lucy's "brain" is one reusable pipeline in `lib/` (`command.ts` router, `ai.ts` + `prompts.ts`, `notion.ts`, `hsk.ts`). Interfaces are three thin adapters — never fork the brain: `app/api/telegram` (bot), `app/api/daily-brief` (cron), and `app/api/chat` + `app/dashboard/` (web). The command layer replies through a `Responder` (`lib/command.ts`) so the same handlers serve Telegram or the web response sink.
- Model routing is role-based and lives in ONE place: `lib/models.ts` (roles → `provider/model-id` slugs, env-overridable via `MODEL_*`) + the `runObject`/`runText` wrappers in `lib/ai.ts` (selection + provider fallback + retry). Providers are called DIRECTLY (no Vercel AI Gateway): `@ai-sdk/deepseek` keyed by `DEEPSEEK_API_KEY` (required; chat/reason/classify), `@ai-sdk/google` keyed by `GOOGLE_GENERATIVE_AI_API_KEY` (optional; long/vision), and `@ai-sdk/openai` keyed by `OPENAI_API_KEY` (optional; vision fallback + embeddings). Construction is lazy + fail-closed per role: chat works on DeepSeek alone, `long` falls back to DeepSeek when Google is unset, `vision` runs Gemini when the Google key is set and falls back to `openai/gpt-4o-mini` when it isn't — it fails closed (throws `NoProviderError`) only when NEITHER Google nor OpenAI is wired, and the Telegram photo path catches that to degrade honestly instead of 500ing. Fallback across providers is done in code (`modelsFor` + `withModelFallback`), not gateway-native. Every model call goes through a wrapper with a `(role, feature)` — don't call `generateObject`/`generateText` directly. `distillEvidence` is the only vision path (must stay multimodal — any vision fallback must be a multimodal model); image bytes are downloaded server-side, never the token-bearing Telegram URL. Live-check structured-output paths with `npm run verify:models` (needs `DEEPSEEK_API_KEY`; add the Google key to also exercise Gemini).
- Semantic retrieval (M3) is a DERIVED, rebuildable read-index in Neon Postgres + pgvector — Notion stays the source of truth and no write path moves off it. `lib/retrieval.ts` owns the one-way sync (chunk + embed changed Notion sources, skip-unchanged by content hash, delete-removed; idempotent) and top-k cosine retrieval; `lib/embeddings.ts` is a DIRECT OpenAI call pinned to `text-embedding-3-small` (1536 dims — MUST match the `vector(1536)` column in `db/schema.sql`); `lib/vector-store.ts` is the `VectorStore` seam (Neon + in-memory impls) so ingestion/ranking are tested without a DB; `lib/db.ts` is the lazy Neon client. Wiring is ADDITIVE + fail-open: `lib/brain.ts` (`buildQuestionBrain`, used by the Telegram + web-chat question paths) prepends retrieved context, and `retrieveContext` returns "" on any problem (no `DATABASE_URL`/`OPENAI_API_KEY`, empty index, unreachable DB) so the bot falls back to Notion-only — never break this. No ORM: schema is raw SQL in `db/schema.sql` (idempotent), applied by `npm run db:migrate`; `npm run sync:index` runs the ingestion. `DATABASE_URL` + `OPENAI_API_KEY` are OPTIONAL in `lib/env.ts` by design (the app boots and runs without them); embeddings fail CLOSED only when actually invoked keyless.
- Web dashboard is single-user, fail-closed: gated by `DASHBOARD_PASSWORD` (session cookie is an HMAC keyed by `CRON_SECRET`, see `lib/auth.ts`). Never expose `/api/chat` or `/dashboard` unauthenticated.
- **Durability invariants — this is a live personal system with no backups; preserve these when touching a write path.** (1) Brain docs are their own only copy: `writeDoc` (`lib/notion.ts`) appends the new blocks *then* deletes the old, and paginates block reads — never reintroduce delete-then-append or a single-page read. (2) A queue task only ever comes back while it is `queued`, so `markActionDone(ok=false)` burns it forever — only failures that can never succeed may burn; transient ones (Anki closed) stay queued and retry (`agent/executor.ts:isTransient`). (3) The watcher is edge-triggered, so a transcript reaches the cloud once: pushes retry, then park in `FAILED_DIR` for replay, and boot rescans recent files (`agent/ingest.ts`). All three are safe because the cloud dedups on content hash (`lessonExists`). (4) Every secret/id in `lib/env.ts` is `.min(1)` — routes compare `Bearer ${SECRET}`, so an empty secret would authorize a bare bearer; `.env.example` must list every required var (a test boots `getEnv()` from it).
- **Telegram is the only inbound surface and it fails SILENTLY when misconfigured — check here first for "the bot ignores every message".** The webhook is registered manually (README "Deploy": `setWebhook?url=.../api/telegram&secret_token=<TELEGRAM_WEBHOOK_SECRET>`). The 401 gate in `app/api/telegram/route.ts` accepts an update only when Telegram's `X-Telegram-Bot-Api-Secret-Token` header equals `TELEGRAM_WEBHOOK_SECRET`; registering `setWebhook` WITHOUT `&secret_token` (Telegram then sends no header) or with a mismatched value drops every update — now `console.warn`-logged so it shows in Vercel logs (`getWebhookInfo` shows the registered URL + whether a secret is set). `secret_token` only allows `[A-Za-z0-9_-]`, so a secret with other chars silently won't register. Second gotcha: EVERY message makes ≥2 sequential model calls (classify in `routeCommand` → distill/answer), so every AI-invoking route MUST set `maxDuration` (telegram/chat=60, daily-brief/ingest=120) or a short platform default kills the function before `sendMessage` — also silent. `getEnv()` runs before the route's try/catch, so any missing required env var 500s the whole route.
